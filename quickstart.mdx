## Overview

This guide walks you through performing your first security scan with SupaScan. You'll extract credentials from a website, analyze the database, and view an interactive report.

<Info>
Before starting, ensure you have [installed SupaScan](/installation).
</Info>

## Basic Scan Workflow

<Steps>
  <Step title="Extract Credentials from Website">
    The simplest way to use SupaScan is to point it at any website using Supabase. It will automatically extract the Supabase URL and anon key:

    ```bash
    supascan -x https://example.com --html
    ```

    This command:
    - Fetches the HTML from `https://example.com`
    - Parses inline scripts and external JavaScript bundles
    - Extracts the Supabase URL and anon key using pattern matching
    - Runs a complete security scan
    - Opens an interactive HTML report in your browser

    <Tip>
    The `-x` or `--extract` flag enables automatic credential extraction from websites.
    </Tip>
  </Step>

  <Step title="View the Analysis Results">
    SupaScan will output progress information as it scans:

    ```bash
    Extracting credentials from https://example.com...
    Found Supabase credentials
    URL: https://xyz.supabase.co
    Analyzing database...
    Discovering schemas...
    Testing RLS policies...
    Generating HTML report...
    ```

    The HTML report opens automatically and includes:
    - **Schema Browser**: Navigate through discovered schemas and tables
    - **Table Explorer**: View sample data from exposed tables with pagination
    - **RPC Tester**: Test discovered RPC functions with parameter forms
    - **Live Query Interface**: Run custom queries against the database
  </Step>

  <Step title="Interpret the Results">
    SupaScan categorizes tables into three types:

    **[+] Exposed Tables**
    ```
    [+] users (~1420 rows exposed)
    ```
    These tables have data accessible with the anon key. This may indicate misconfigured RLS policies.

    **[-] Protected Tables**
    ```
    [-] sessions (0 rows - empty or RLS)
    ```
    These tables exist but return no data, indicating proper RLS protection or empty tables.

    **[X] Denied Tables**
    ```
    [X] admin_users (access denied)
    ```
    These tables deny read access entirely - the most secure configuration.
  </Step>
</Steps>

## Alternative Methods

### Using Manual Credentials

If you already have the Supabase URL and anon key, provide them directly:

```bash
supascan --url https://xyz.supabase.co --key eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... --html
```

<Note>
The anon key is a JWT token that starts with `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9`.
</Note>

### Console Output Only

For terminal output without the HTML report, omit the `--html` flag:

```bash
supascan --extract https://example.com
```

## Advanced Usage

### Analyze Specific Schema

Focus analysis on a specific schema:

```bash
supascan -x https://example.com --schema public
```

### Dump Table Data

Extract data from a specific table:

```bash
supascan --extract https://example.com --dump public.users --limit 100
```

<Warning>
Be cautious when dumping data. Even with the anon key, exposed data may contain sensitive information.
</Warning>

### Call RPC Functions

Test discovered RPC functions:

```bash
supascan --extract https://example.com --rpc public.search_users --args '{"query": "admin"}'
```

You can use environment variables in arguments:

```bash
export USER_ID="123"
supascan --url $URL --key $KEY --rpc public.lookup --args '{"id": "$USER_ID"}'
```

### JSON Output

Generate machine-readable JSON output for integration with other tools:

```bash
supascan --extract https://example.com --json > report.json
```

<Tip>
JSON output is useful for CI/CD pipelines or automated security scanning workflows.
</Tip>

## What to Look For

When reviewing scan results, pay attention to:

1. **High Row Counts on Exposed Tables**: Tables with thousands of exposed rows likely need RLS policies
2. **Sensitive Table Names**: Tables named `users`, `profiles`, `private_*` should be protected
3. **RPC Functions**: Verify that RPC functions don't expose sensitive operations
4. **Schema Information**: Exposed column names may reveal application logic
